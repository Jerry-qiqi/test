> AI Web 端从空目录起步，一周内交付 MVP 版本——SSE 流式传输层、AI 聊天 UI（全页面 + 浮动挂件双模式）、业务卡片支持，全部就位。这篇文章分享我们在这个过程中沉淀出的 AI 工作流：**Ray**——一条 10 个 Agent 组成的 Spec-Driven 自治开发流水线。

## 我们遇到的问题

AI 编程助手大家都在用了。但我们在实际开发中发现，大多数人用它的方式是——**让 AI 帮我写一段代码**。

问题在于：代码只是研发链路中的一环。需求澄清、类型设计、测试编写、质量审计、文档沉淀……这些环节仍然靠人手动衔接。AI 写完代码，人要手动补文档、手动跑测试、手动检查有没有偏离需求。



我们的做法是：**把整条研发链路变成一条 Spec-Driven 的自治流水线**，命名为 **Ray**（光线）。Ray 基于 Claude Code 的 Skill 机制构建——每个 Agent 是一个独立的 Skill 定义，通过 plugin 命名空间统一管理。10 个专业 Agent 各司其职，需求进去，生产级代码 + 活文档出来。人的角色从"写代码"变成"描述需求、验收结果"。 &#x20;

## Ray 的核心优势

1. **一句需求进去，代码+文档+测试全出来** — 需求文档、类型合约、测试用例、实现代码、产品文档，全链路自动生成

2. **10 个专职 Agent，7 步自动流转** — 需求、架构、测试、实现、审计、文档各有专职 Agent，零人工衔接

3) **300+ 次迭代后，AI 仍精确知道项目全貌** — 每次迭代自动沉淀为活文档，人会离职，文档不会

4) **质量靠流水线硬卡，不靠 code review** — 测试先于代码，审计 100 分制量化到行号，不达标自动打回

5. **开发者从写代码的人，变成定义产品行为的人** — 你说"点击后跳转订单页"，不用写 `router.push`

## Spec-Driven：规格驱动一切

> 详细文档：[Spec-Driven Development](https://gtglobal.jp.larksuite.com/wiki/B5kZwnLQUiZEGlkW40Xj9PFOpDc)

Ray 的核心方法论是 **Spec-Driven Development**——所有环节围绕"规格"运转，而不是围绕代码。

```plain&#x20;text
传统模式：开发者写代码 → 事后补文档 → 文档很快过时
我们的模式：规格先行 → 代码从规格生成 → 文档从规格沉淀 → 规格永远是真相
```

具体来说，Ray 中每个 Agent 都在消费或生产规格：

```plain&#x20;text
ray:trace     产出 → 需求规格（场景、验收标准、Non-Goals）
ray:architect 产出 → 类型合约（数据结构、状态机、不变式）
ray:qa        消费 ← 类型合约，产出 → 测试规格（TDD RED 态）
ray:coder     消费 ← 类型合约 + 测试规格，产出 → 实现代码
ray:audit     消费 ← 需求规格 + 类型合约 + 测试 + 代码，产出 → 审计报告
ray:update-map消费 ← 全部产物，产出 → 活文档（下次迭代的上下文）
```



**规格是 Agent 之间的通信协议。** 没有规格，Agent 之间只能靠"猜"对方的意图。有了规格，三个下游 Agent（QA、Coder、Audit）共用同一套类型定义，分歧就消失了。



这也引出了我们的一条铁律：**尽量避免人工直接修改代码。**



### 为什么不建议手动改代码



这是推行时最常被质疑的点："为什么不直接改代码？"



先看代价对比：

|           | 手动改代码                        | 回到 ray:trace 补需求       |
| --------- | ---------------------------- | ---------------------- |
| **当下**    | 快，3 分钟改完                     | 慢，要描述清楚约束              |
| **下一轮**   | Spec 和代码脱节 → 审计失败 → 重新对齐     | 全链路自动同步                |
| **第 N 轮** | AI 基于错误上下文生成代码 → 逻辑漂移 → 代码腐化 | 活文档永远是最新的，AI 每次读到的都是真相 |



手动改代码的问题不在"这一次"，在于**它破坏了 Spec 和代码之间的绑定关系**。一旦绑定断了，后续每次迭代 AI 都在错误的上下文上生成代码——这就是"代码腐化"的起点。



**我们不是剥夺你改代码的权力，而是赋予你"逻辑豁免权"。** 你在更高维度定义"应该怎样"，重复性工作让`ray:coder` 去自愈循环。你说"这个按钮点击后应该跳转到订单页"，而不是去写 `router.push('/order')`。

## Ray 的 10 个 Agent



```plain&#x20;text
ray:trace → ray:pipeline → ray:architect → ray:qa → ray:coder → ray:audit → ray:update-map
                                                                                    ↑
                                                              ray:origin  ray:explore  ray:digest
```

| Agent              | 角色    | 一句话                              |
| ------------------ | ----- | -------------------------------- |
| **ray:trace**      | 需求入口  | 把模糊的人类需求变成可执行的规格文档               |
| **ray:pipeline**   | 总调度器  | 按任务复杂度自动分级，决定走哪些步骤               |
| **ray:architect**  | 合约架构师 | 生成类型合约——下游所有 Agent 的共享语言         |
| **ray:qa**         | 测试工程师 | 代码不存在时就写测试，定义验收边界                |
| **ray:coder**      | 实现工程师 | 自愈循环：跑测试 → 失败 → 修代码 → 再跑，最多 10 轮 |
| **ray:audit**      | 质量审计  | 100 分制评分，不变式违反一票否决               |
| **ray:update-map** | 知识沉淀  | 每次迭代后更新活文档，成为下次需求的上下文            |
| **ray:origin**     | 文档初始化 | 首次扫描代码库，从零生成活文档体系                |
| **ray:explore**    | 项目导览  | 2 分钟建立全局视野，纯只读                   |
| **ray:digest**     | 文档归纳  | 定期将碎片化记录归纳为精炼的产品手册               |

## 流水线怎么跑

### 自适应分级

我们在实践中发现，不是每个任务都需要走全套。改个按钮颜色走完整 TDD 是浪费。所以 `ray:pipeline` 在启动时自动判断任务级别：

| 级别           | 触发条件              | 执行步骤               | 跳过        |
| ------------ | ----------------- | ------------------ | --------- |
| **PATCH**    | bug 修复、样式调整、≤5 文件 | 实现 → 轻量审计 → 知识沉淀   | 合约设计、预写测试 |
| **FEATURE**  | 新功能、新 API、新组件     | 完整 Spec-Driven TDD | 无         |
| **REFACTOR** | 结构变更、不改行为         | 实现 → 轻量审计 → 知识沉淀   | 合约设计、预写测试 |

改个颜色 5 分钟搞定。新功能走完整 TDD——类型先行、测试先行。da

### FEATURE 完整流程

### 各 Agent 做了什么

#### **ray:trace — 需求入口**

四步把模糊需求变成可执行规格：

1. **影响分析**：读产品地图 → 定位受影响的模块和组件 → 识别组件级别 → 不变式不得违反

2. **澄清歧义**：逐个提问，优先多选题

3) **起草需求文档**：必含"当前状态（Before）"、场景化用户故事、Non-Goals、Open Issues

4) **用户确认**：写入 trace 文件，进入 pipeline

#### **ray:architect — 合约设计**

输出**类型合约文件**，是流水线的"宪法"。按组件级别生成不同深度：

* **轻量**：数据结构、API 端点

* **标准**：+ 状态列表、边界情况

* **完整**：+ 不变式、状态机、故障旅程

#### **ray:qa — 测试工程师**

先写测试，必须先失败（RED）。如果有测试"无需实现就通过"，说明规范有问题，标记上报。

#### **ray:coder — 实现工程师**

自愈循环：跑测试 → 全绿退出 → 否则分析失败原因 → 只改 src/ → 再跑（最多 10 轮）。铁律：不能改测试、不能改 spec、发现安全漏洞立即停止。

#### **ray:audit — 质量审计**

100 分制，7 个维度（需求一致性 25 / 安全性 20 / OpenSpec 合规 15 / 代码质量 15 / 分级合约合规 10 / 产品文档就绪度 10 / 治理合规 5）。拒绝时精确到**文件名 + 行号 + 问题 + 修法**。

#### **ray:update-map — 知识沉淀**

更新 trace frontmatter → 重新生成 MAP.md → 更新产品文档网络 → 写 ADR。

## AI 工作流需要哪些文档



在我们的 Spec-Driven 工作流中，**文档是 Agent 的输入和输出——没有文档，Agent 就无从下手。**



AI Agent 每次启动都是空白的。项目越大，"记住全貌"越不可能。我们的解法：**把知识存进文件，让文件成为项目的长期记忆。**



Ray 需要四类文档，分为**流转文档**和**沉淀文档**两层：

| 文档类型            | 性质               | 谁写                                       | 谁读                                  |
| --------------- | ---------------- | ---------------------------------------- | ----------------------------------- |
| **需求追溯（Trace）** | 碎片，每次迭代一个        | ray:trace（人确认）                           | pipeline、architect、audit            |
| **类型合约（Spec）**  | 碎片，每次迭代一个        | ray:architect                            | qa、coder、audit                      |
| **技术索引（MAP）**   | 轻量索引，自动生成        | ray:update-map                           | pipeline（快速定位）                      |
| **产品文档网络**      | **精炼沉淀，三层 Wiki** | ray:update-map / ray:origin / ray:digest | **ray:trace （影响分析）、pipeline、人（审阅）** |



关键设计：**产品文档网络才是 Agent 做影响分析时的主要上下文来源，不是 MAP.md。**



MAP.md 是轻量的碎片索引（每个 FEAT 一行），用于快速定位"这个需求影响哪个域"。但真正告诉 Agent"改之前是什么样"的，是产品文档网络——它是 trace 碎片经过归纳后的精炼产物。



**每次迭代产生碎片（Trace + Spec），沉淀为精炼文档（产品文档网络）。** 项目做到第 300 个 FEAT 时，`ray:trace` 不需要读 300 个 trace 文件——它读产品文档就知道"改之前是什么样"。

## 产品文档网络：为什么这样设计



### 问题



项目做了 300+ 个 FEAT 后，知识散落在：

* 几百个 trace 文件（碎片化的需求记录）

* 代码注释（和代码耦合，重构就丢了）

AI Agent 每次启动要从零理解项目。让它读几百个 trace——context window 爆了。只读代码——它不知道"为什么这样设计"。

### 设计：三层 Wiki

```plain&#x20;text
docs/product/
├── PRODUCT-MAP.md                        ← 决策层（≤60 行）
├── modules/
│   ├── ai-chat/
│   │   ├── index.md                      ← 叙事层（用户旅程）
│   │   ├── ChatContainer.md              ← 规约层（完整级）
│   │   ├── Sidebar.md                    ← 规约层（标准级）
│   │   └── EmptyState.md                 ← 规约层（轻量级）
│   ├── cards/
│   │   ├── index.md
│   │   ├── SwapCard.md
│   │   └── ...
```

| 层       | 文件                            | 回答什么问题         | 目标大小     |
| ------- | ----------------------------- | -------------- | -------- |
| **决策层** | PRODUCT-MAP.md                | 产品有哪些模块？       | ≤60 行    |
| **叙事层** | modules/{name}/index.md       | 用户怎么用这个模块？     | 每模块 1 文件 |
| **规约层** | modules/{name}/{Component}.md | 这个组件的确定性边界是什么？ | 每组件 1 文件 |

### 为什么三层

**为了让 Agent 按需加载，而不是一次读完。**

Agent 做影响分析时的读取路径：

```plain&#x20;text
1. 读 PRODUCT-MAP.md（60 行）          → 知道有哪些模块
2. 只读受影响模块的 index.md            → 知道用户旅程和组件列表
3. 只读受影响组件的 Component.md        → 知道交互、状态、不变式
```

三步下来，Agent 只消耗了几百行 token，却精确知道"改之前是什么样"。如果把所有信息塞进一个大文件，要么太大读不完，要么太粗定位不准。

### 组件分级

不是所有组件都需要同样深度的文档。一个纯展示的 EmptyState 和一个涉及 SSE 流式传输的 ChatContainer 复杂度完全不同。

| 级别     | 判断条件        | 文档包含                       |
| ------ | ----------- | -------------------------- |
| **轻量** | 无状态、纯展示     | 功能、界面结构、交互表                |
| **标准** | 有状态但简单      | + 状态列表、边界情况、Non-Goals      |
| **完整** | 复杂状态流转、网络交互 | + **不变式**、**状态机**、**故障旅程** |



完整级的三个额外维度，直接服务于下游 Agent：

* **不变式** → `ray:architect` 将其转化为类型约束，`ray:coder` 在所有路径上保证其成立

* **状态机** → `ray:qa` 为每个状态转换写测试

* **故障旅程** → `ray:coder` 必须实现降级策略和恢复路径

### 组件间关系



组件不是孤立的。ChatContainer 依赖 Sidebar，Sidebar 触发 ChatContainer 切换会话。这些关系用**双向链接**维护：



```markdown
## 关系

| 方向 | 类型 | 目标 | 说明 |
|------|------|------|------|
| →   | 嵌入 | [Sidebar](Sidebar.md) | 渲染侧边栏 |
| →   | 依赖 | [SSETransport](../transport/SSETransport.md) | 流式传输 |
| ←   | 被依赖 | [FloatingAiChat](../floating/FloatingAiChat.md) | 浮窗模式复用 |
```



铁律：A→B 关系写入时，必须同时在 B 的文件中添加 ←A。**禁止孤立链接。**

## 人在哪里协作

Ray 不是"全自动不需要人"。人的角色是**决策者和验收者**，Agent 是执行者。



**四个人工参与点：**



1. **需求确认**（ray:trace 输出后）— 你同意这份需求文档吗？

2. **分级确认**（ray:pipeline Phase 0）— 这个任务是 PATCH 还是 FEATURE？

3) **最终验收**（ray:update-map 之后）— 看看结果，满意吗？

4) **产品文档审计**（持续）— 三层 Wiki 就是写给人看的，人要定期审阅，确保它准确反映产品现状

其余环节全自动。如果审计硬拒（<60 分），流水线停止，人介入决策。



**为什么人要审计产品文档？** Agent 生成的文档基于代码和 trace 推断，但它不能判断"这个交互是否符合产品意图"。产品文档网络之所以设计成三层 Wiki 而不是机器内部格式，就是为了让人能直接阅读、审阅、纠偏。文档对了，后续每次 `ray:trace` 读到的上下文就对了——这是整条链的根基。

### 协作原则

* **人描述意图，不描述实现** — 告诉 Ray "用户点击后应该跳转到订单页"，而不是"在 onClick 里调用 router.push('/order')"

* **人通过需求修正 AI，不通过直接改代码** — 代码不对？回到 `ray:trace` 补充约束

* **人做不确定性决策，AI 做确定性执行** — "要不要支持 RTL？"是人的决策。"如何实现 RTL"是 AI 的执行。
